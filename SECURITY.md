# Security Policy

## Supported scope

This policy applies to security issues in Food Run Community Edition (CE) that are reported in good faith and relate to:
- source code in this repository
- build or runtime behavior directly controlled by this repository
- dependency or configuration issues that materially affect this repository

This policy does **not** guarantee support for:
- forks not maintained by the Project Owner or maintainers
- unofficial deployments
- downstream packaging outside official project control
- out-of-scope social engineering or physical attacks unrelated to the software itself

## Reporting a vulnerability

Please **do not** open public issues, pull requests, or discussions for suspected security vulnerabilities.

Instead, report vulnerabilities privately to the project using the contact method published by the Project Owner.

### Preferred contact path

Until a dedicated security mailbox is published, use one of the following private channels if available:
- the official repository owner or organization contact email
- the official project website contact form
- a private maintainer contact method explicitly published by the Project Owner

## What to include in a report

Please include, where possible:
- a clear summary of the issue
- affected component, file, route, or feature
- impact and severity as you understand it
- reproduction steps
- proof of concept, if safe to share
- affected versions, branches, or environments
- any suggested mitigation or patch direction

## What to expect from us

Maintainers may:
- acknowledge receipt
- ask follow-up questions
- reproduce and validate the issue
- prioritize and prepare a fix
- coordinate disclosure timing
- decline to publish exploit details before a fix is available

Response times are best-effort and may vary depending on project capacity and severity.

## Disclosure approach

Please allow maintainers a reasonable opportunity to investigate and remediate the issue before public disclosure.

If a vulnerability is confirmed, maintainers may choose to:
- patch privately first, then disclose
- publish a security advisory
- release a fix before sharing technical details
- withhold exploit details that would materially increase risk to users

## Safe harbor

If you act in good faith, avoid privacy violations, avoid service disruption, avoid data destruction, and do not exploit the issue beyond what is reasonably necessary to demonstrate it, the project will treat your report as a good-faith security disclosure.

## No bounty

Unless explicitly stated elsewhere, this project does not currently offer a bug bounty or guaranteed compensation for vulnerability reports.
