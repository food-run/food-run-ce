# Master

## One-Line Vision

Food Run should feel like elegant minimalist execution built on a playful-comic token system.

## Product Posture

Food Run is a workflow product, not a dashboard product.

Its job is to reduce decision load across a recurring planning cycle:
1. Bring recipes in
2. Clean or confirm what matters
3. Plan meals
4. Generate a reliable shopping outcome

The interface should support that flow with calm clarity and obvious next steps.

## Visual Character

Food Run should feel:
- Warm
- Clean
- Expressive in controlled ways
- Human
- Approachable
- Trustworthy
- Polished
- Efficient

It should not feel:
- Sterile
- Corporate-generic
- Childish
- Loud
- Overly gamified
- Cluttered
- Fake-smart

## Core Design Laws

### 1) One Dominant Job per Screen

Each screen or major region should have one dominant job.

If the surface seems to be doing many things at once, reduce it.

### 2) Hierarchy Before Decoration

Users should know where to look and what to do before they notice style details.

If a surface only works once colors, borders, or flourishes are added, the structure is weak.

### 3) Restraint Creates Quality

Minimalist or refined designs need restraint, precision, and careful attention to:
- Spacing
- Typography
- Alignment
- Rhythm
- Contrast
- State styling
- Copy discipline

Quiet interfaces are not easier to design. They are less forgiving.

### 4) Playful Does Not Mean Chaotic

The playful-comic token system gives Food Run warmth and personality.

That warmth must be controlled through:
- Readable type usage
- Disciplined spacing
- Restrained color emphasis
- Consistent component behavior
- Subtle motion
- Honest writing

### 5) Clarity Is the Brand

Food Run should become memorable because it feels easy, calm, and trustworthy.

The brand impression should come from:
- Obvious flow
- Clean layout
- Good language
- Visual discipline
- Subtle charm

Not from novelty for its own sake.

### 6) Honesty Beats Spectacle

Do not imply product maturity that does not exist.

Live users will trust grounded clarity more than flashy overclaiming.

### 7) Surface Count Matters

The current reviewer shell carries more boxed surfaces than the refined design system should preserve for Sprint 1.

Every bordered panel, muted container, empty-state box, and card-like wrapper must justify itself.

If spacing, rhythm, and heading structure can do the job, prefer that over another bordered surface.

## Layout Doctrine

- Prefer clean sectional composition
- Prefer whitespace over excessive containers
- Prefer one strong primary region over many equal regions
- Prefer top-to-bottom guidance
- Prefer local grouping that can be understood at a glance
- Prefer fewer borders and fewer nested surfaces
- Prefer a stable shell that makes page transitions feel predictable

## Typography Doctrine

- Typography is a core part of the product’s personality
- The comic influence should feel warm and distinctive, not messy
- Prioritize legibility, rhythm, and stable role distinctions
- Use fewer sizes with stronger contrast between roles
- Ensure buttons, labels, headings, and body text each have a clear job

## Color Doctrine

- Use color semantically
- Accents should guide, not shout
- Neutrals should carry most of the structure
- State colors must remain readable and believable
- Dark mode should preserve clarity and warmth, not simply invert values

## Interaction Doctrine

- Every interactive control should have intentional states
- Motion should confirm action or orientation, never distract
- Focus-visible should be strong and attractive
- Disabled states should be understandable, not merely faded
- Loading states should remain calm and honest

## Content Doctrine

- Labels should be direct
- Buttons should describe the result, not use generic verbs
- Helper text should reduce uncertainty
- Errors should explain what happened and what to do next
- Empty states should orient the user toward the next useful action

## Trust Doctrine

Food Run should feel grounded.

That means:
- No fake certainty
- No fake automation sophistication
- No fake data persistence
- No fake completeness
- No jargon used to hide product limitations

## Review Doctrine

A screen should not be considered good because it looks modern.

A screen is good when:
- The dominant job is obvious
- The next step is obvious
- The hierarchy is clean
- The user does not have to guess
- The copy is specific
- The states are believable
- The spacing and typography feel intentional
- The page works on mobile and desktop
- It still feels like Food Run
- It is not relying on leftover prototype card patterns to create structure

## Required Questions Before Implementation

1. What is the user trying to do here?
2. What must they understand first?
3. What is the one primary action?
4. What should be visually quiet?
5. What can be removed?
6. What state is actually real today?
7. Which surfaces can be simplified, softened, or removed?
8. What style choices support clarity instead of noise?

## Required Questions Before Approval

1. Is the dominant job obvious in three seconds?
2. Does the page feel calmer than before?
3. Does the page overclaim anything?
4. Are spacing and typography doing enough work?
5. Does the playful-comic system still feel disciplined?
6. Are interactive states complete enough for live users?
7. Would a first-time user understand the next step without explanation?
8. Did we remove or reduce any leftover prototype card artifacts that no longer help the user?
