# Shopping

## Page Role

The Shopping page is the consolidated outcome surface.

It should help users trust what they need to buy and move through that information efficiently.

## Dominant Job

Review and use a consolidated shopping output with confidence.

## Primary Priorities

- Scanability
- Trust
- Consolidation clarity
- Checkability
- Low-friction use in real-world shopping contexts

## Layout Guidance

- Prioritize the list itself
- Keep grouping logic obvious
- Make the most important structure visible before secondary details
- Keep support actions subordinate to the list outcome

## Content Guidance

Shopping content should feel:
- Clear
- Structured
- Dependable
- Practical

Avoid turning the list into a data dump or an over-designed productivity board.

## Trust Notes

Do not imply:
- Perfect deduplication quality if it is not guaranteed
- Inventory intelligence that does not exist
- Purchasing optimization that does not exist

If the output has limitations, the UI should remain honest without undermining usability.

## Interaction Guidance

Where interactive:
- Keep marking or updating actions obvious
- Preserve row clarity
- Do not bury important list state in decorative styling

## Surface Guidance

The current reviewer shell uses a boxed empty-state treatment for Shopping.

Sprint 1 should evaluate whether that message still needs a full surfaced container.

Prefer the lightest treatment that still preserves:
- Readability
- Trust
- Clear next steps

The current reviewer shell should keep the empty state practical and lightly framed so the eventual list can remain the dominant structure.

## Mobile Behavior

This page is especially important on mobile.

Prioritize:
- Row readability
- Tap comfort
- Stable grouping
- Quick scanning in real-world use

## Banned Clutter

Do not add:
- Oversized explanatory panels above the list
- Unnecessary decorative containers
- Too many row badges
- Overly dense metadata before core item text

## Review Questions

1. Does the list feel trustworthy?
2. Can users scan it quickly?
3. Is grouping logic obvious?
4. Does the page support practical shopping use instead of merely looking organized?
5. Is the empty-state or list framing lighter and more disciplined than the old prototype carryover?
