# Planner

## Page Role

The Planner is where users organize meals over time.

This surface must balance structure with flexibility.

## Dominant Job

Plan meals clearly without feeling overwhelmed by too much simultaneous context.

## Primary Priorities

- Sequence and structure
- Obvious planning actions
- Clear day or meal grouping
- Manageable context
- Low-friction updates

## Layout Guidance

- Emphasize the planning structure first
- Make each planning region easy to understand at a glance
- Keep supporting recipe details subordinate
- Avoid overloading the Planner with side tools unless clearly necessary

## Experience Goal

The Planner should feel:
- Orderly
- Calm
- Easy to edit
- Easy to revisit
- Stable across a weekly rhythm

It should not feel:
- Crowded
- Board-game-like unless the structure truly demands it
- Analytics-heavy
- Visually fragmented

## Interaction Guidance

Planning interactions should:
- Feel reversible
- Feel local and understandable
- Reveal state changes clearly
- Avoid hidden complexity

## Surface Guidance

The current reviewer shell uses a boxed empty-state treatment inside the routed page.

Sprint 1 should not blindly preserve that pattern.

If the empty state can read clearly with:
- Stronger spacing
- Better heading hierarchy
- Cleaner copy
- A lighter visual container

then prefer that over a full card-style message.

The current reviewer shell should treat this empty state as orientation copy first, not as a featured panel.

## Trust Notes

Do not imply:
- Advanced scheduling intelligence that does not exist
- Conflict detection or optimization that does not exist
- Automatic coordination sophistication that does not exist

## Mobile Behavior

On mobile:
- Preserve plan structure clearly
- Avoid over-compressing day or meal groupings
- Prioritize legible sequencing over dense summary views
- Keep editing actions nearby and understandable

## Banned Clutter

Do not add:
- Too many simultaneous control layers
- Decorative swim lanes with weak meaning
- Overwhelming metadata
- Competing call-to-action rows in each planning group

## Review Questions

1. Can users understand the plan structure quickly?
2. Is the next planning action obvious?
3. Does the page feel manageable rather than heavy?
4. Does the structure support repeated weekly use?
5. Is the empty-state treatment lighter and more disciplined than old prototype card patterns?
