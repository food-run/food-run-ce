# Product Lens

## Purpose

This file translates general design principles into Food Run-specific product behavior.

The design system should not float above the product. It should support Food Run’s real workflow and current product maturity.

## What Food Run Is

Food Run is a meal-planning and shopping workflow product.

Its value is not raw data display. Its value is reducing friction between:
- Collecting recipes
- Preparing to cook
- Planning meals
- Producing a trustworthy shopping outcome

## What Users Need Most

Users need to feel:
- They can get started quickly
- The system is not wasting their time
- The next step is obvious
- Mistakes can be corrected without drama
- The interface is helping them think clearly

## Core Product Behaviors the UI Should Support

### 1) Low-Friction Intake

The import and collection flow should feel easy to start.

The interface should reduce hesitation and make the first action obvious.

### 2) Guided Clarification

Where the product asks the user to review, confirm, or adjust data, the UI should feel calm and understandable.

The system should not sound smarter than it is.

### 3) Repeatable Planning

Planning should feel orderly and stable, not like a crowded task board unless that is truly necessary.

The user should feel they can make progress incrementally.

### 4) Reliable Outcome Generation

The shopping result should feel dependable and easy to scan.

The system should emphasize confidence through structure and clarity, not hype.

## Trust Posture

Food Run should feel honest at all times.

That means:
- No fake AI sophistication
- No fake confidence metrics
- No hidden assumptions presented as certainty
- No misleading status labels
- No implied background processing that is not real
- No pretending a static reviewer shell is fully powered

## Personality Posture

Food Run can have warmth and lightness.

That warmth should come from:
- Word choice
- Type character
- Spacing generosity
- Rounded friendliness
- Small moments of charm

It should not come from:
- Visual clutter
- Too many colors
- Noisy iconography
- Meme-like copy
- Cartoon overload

## Workflow Posture

Food Run is cyclical and repeatable.

The UI should therefore feel:
- Steady
- Easy to revisit
- Not over-optimized for one-time wow moments
- Forgiving of incremental use

## Surface Posture

The UI should not make every workflow step feel like a stack of cards.

Food Run should prefer:
- Cleaner sections
- Better rhythm
- Clear headings
- Soft grouping
- Fewer repeated bordered boxes

Use boxed surfaces only when they improve:
- Local scope
- Repeated-item clarity
- Scannability
- Action grouping

## Page Priorities by Route

### Import
Priority is confidence and low friction.

### Recipes
Priority is scanability and selection.

### Planner
Priority is clarity, sequencing, and manageable context.

### Shopping
Priority is trust, consolidation, and checkable output.

## What Success Should Feel Like

A successful Food Run interface should make the user think:
- This looks easy
- I know where to start
- This feels organized
- This feels trustworthy
- I can come back to this weekly

## Food Run-Specific Review Questions

1. Does this surface reduce decision load?
2. Does this page make the next step feel obvious?
3. Does the page imply capabilities we do not yet have?
4. Does the personality help or distract?
5. Is the page calm enough for recurring use?
6. Would a user trust the output without feeling oversold?
7. Are there any leftover card artifacts that are adding weight without adding clarity?
