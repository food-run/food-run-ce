# Principles

## Purpose

This file defines the first-principles method for designing or revising Food Run interfaces.

The goal is to avoid style-first UI work and instead build from:
- User intent
- Information needs
- Action priority
- Trust requirements
- Product truth

## Method

### Step 1: Define the Dominant Job

State the main thing the user is trying to accomplish.

Use one sentence.

Examples:
- Import a recipe URL and understand what happens next
- Review available recipes and choose one to plan
- Assemble a week’s meals without losing context
- Confirm a shopping outcome with confidence

If more than one dominant job appears, the page probably needs reduction or clearer sectional hierarchy.

### Step 2: Define the Minimum Information Needed Before Action

Ask:
- What must the user know before acting?
- What can be deferred until after the action?
- What can be hidden or softened?
- What belongs in helper text instead of persistent body copy?

The answer is usually less than the first draft suggests.

### Step 3: Identify the Primary Action

The page should make one action unmistakably primary.

Primary does not only mean visually larger. It also means:
- Better positioned
- More specifically labeled
- More strongly supported by the surrounding layout
- Not competing with equal alternatives

If there are many primary actions, there is no primary action.

### Step 4: Identify What Should Stay Quiet

Not every element should compete for attention.

Quiet elements often include:
- Secondary navigation
- Descriptive support text
- Footnotes
- Status indicators that are not urgent
- Non-blocking metadata
- Optional tools

Quiet does not mean hidden. It means subordinate.

### Step 5: Reduce the Structure

Before styling:
- Remove duplicate explanation
- Merge repeated labels
- Flatten unnecessary nesting
- Remove decorative wrappers
- Simplify call-to-action sets
- Shorten copy where meaning is preserved
- Remove boxed surfaces that exist only from old card habits

Ask:
What would remain if this page had to become 30 percent simpler?

### Step 6: Prove the Hierarchy

Use the squint test:
- What stands out first?
- What stands out second?
- Does that match the intended user journey?

Use the scan test:
- Can a user find the main action without reading every word?
- Do section boundaries feel obvious?
- Are labels informative enough on their own?

If not, adjust structure before adding more style.

### Step 7: Style With Restraint

Only after the above steps:
- Apply spacing rhythm
- Refine type roles
- Define surface contrast
- Tune color emphasis
- Refine states
- Add subtle personality cues

Style should confirm the structure, not rescue it.

### Step 8: Check Against Product Truth

Make sure the page does not imply:
- Stronger automation than exists
- Stronger confidence than exists
- Stronger persistence than exists
- Stronger collaboration than exists

Honest design builds durable trust.

## Composition Rules

### Prefer
- Clear vertical flow
- Strong section headers
- One obvious input region
- One obvious action region
- Whitespace as grouping
- Short, direct copy
- Readable type scale
- Semantic color usage
- Fewer surfaces with clearer purpose

### Avoid
- Card piles
- Equal emphasis everywhere
- Decorative icon repetition
- Competing call-to-action rows
- Large introductory paragraphs
- Weak-contrast labels
- Placeholder-only forms
- Motion as decoration

## Minimalist Precision Checklist

When the design is intentionally minimal, check:
- Is spacing carrying enough hierarchy?
- Is typography distinct enough by role?
- Are quiet elements still readable?
- Do controls feel precise rather than bland?
- Are alignments clean?
- Do surfaces feel deliberate, not empty?
- Does the page still have character?
- Is there a bordered container that no longer needs to exist?

## Page-Level Decision Order

When choosing what to change, use this order:
1. Structure
2. Copy
3. Hierarchy
4. Spacing
5. Typography
6. Color and surface treatment
7. Interaction and motion

This order prevents polish work from masking structural problems.

## Success Criteria

A page designed from first principles should:
- Make the user job obvious
- Reduce unnecessary thought
- Preserve trust
- Feel calm under repeated use
- Maintain Food Run’s playful warmth in a disciplined way
- Depend less on leftover prototype card framing
